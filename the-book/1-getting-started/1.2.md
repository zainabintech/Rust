# 1.2 — Hello, World!

## Commands

- `rustc main.rs` — compile a Rust source file.
- `.\main` (Windows) or `./main` (Linux/macOS) — run the executable that was produced.

## File conventions

- Rust files always end with the `.rs` extension.
- If you're using more than one word in your filename, the convention is to use an underscore to separate them.
  - Example: `hello_world.rs`, **not** `helloworld.rs`.

## The Hello World program

```rust
fn main() {
    println!("Hello, world!");
}
```

### Breakdown

**The function definition**

```rust
fn main() {
}
```

- These lines define a function named `main`. The `main` function is special: it is always the first code that runs in every executable Rust program.
- **Every executable Rust program must contain a function named `main`.**
  - In simple words: `main` is the *starting point* of the program. When you run your Rust program, Rust looks for a function called `main` and starts running the code inside it. Without a `main` function, Rust doesn't know where to begin, so the program won't run.
- The first line declares a function named `main` that has no parameters and returns nothing. If there were parameters, they would go inside the parentheses `()`.
- The function body is wrapped in `{}`.
- Rust requires curly brackets around all function bodies. It's good style to place the opening curly bracket on the same line as the function declaration, with one space in between.

**Formatting tool**

There is an automatic formatter called `rustfmt` that formats the code in a particular style. It is already installed on the computer. (TBH I will avoid using it for now, since I'm here to learn and want my brain to get used to writing it without hints.)

**The body**

```rust
println!("Hello, world!");
```

This prints the string to the screen. Three important things to notice:

1. `println!` calls a Rust **macro**. If it had called a function instead, it would be entered as `println` (without the `!`).
2. `"Hello, world!"` is a **string**. We pass this string as an **argument** to `println!`, so that the string can be printed to the screen.
3. We end the line with a **semicolon** `;`, which indicates that this expression is over and that the next one is ready to begin. Most lines of Rust code end with a semicolon.

### Symbols cheat-sheet

| Symbol | Meaning |
|---|---|
| `{}` | curly brackets — wrap function bodies |
| `!` | calling a macro instead of a normal function |
| `;` | semicolon — ends an expression |

## Compiling and running

Before running a Rust program, you must **compile** it using the Rust compiler with the `rustc` command, passing it the name of the source file:

```
rustc main.rs
```

After compilation is successful, Rust outputs a **binary executable**.

> **Quick check:** Say you run `rustc hello.rs` from the command line. What happens next?
> - `rustc` generates a binary executable named `hello`.
> - Running `rustc` **checks and compiles** your program, but does **not** execute it. To actually run the program, you have to invoke the executable yourself (e.g. `.\hello` on Windows, `./hello` on Linux/macOS).

```
PS D:\Rust\projects\hello_world> ls

    Directory: D:\Rust\projects\hello_world

Mode                 LastWriteTime         Length Name
----                 -------------         ------ ----
-a----          5/8/2026   5:48 PM         128512 main.exe
-a----          5/8/2026   5:48 PM        1257472 main.pdb
-a----          5/8/2026   5:47 PM             88 main.rs
```

This shows:
- the source code file `main.rs`,
- the executable file `main.exe` (on Windows),
- a file containing debugging information `main.pdb` (also on Windows).

To run the executable:

```
.\main
```

## Compiled vs interpreted languages

- Rust is an **ahead-of-time (AOT) compiled language**: you can compile a program and give the executable to someone else, and they can run it even without having Rust installed.
- In Python, Ruby, or JavaScript, if you give someone a `.py`, `.rb`, or `.js` file, they need Python / Ruby / Node installed to run it.
- In those interpreted languages you only need one command to run your program — but you also create a runtime dependency.
- Everything is a tradeoff in language design.

## Looking ahead

Compiling with `rustc` is fine for simple programs, but as the project grows, you'll want to manage all the options and make it easy to share your code. That's where **Cargo** comes in — it will help you write real-world Rust programs.

## Questions & Doubts (cleared)

### 1. What's the difference between PowerShell and CMD?
Both are command-line shells on Windows. CMD is the older one (basic batch commands). PowerShell is newer, more powerful, and works with structured objects instead of just plain text — you can pipe results between commands and use cmdlets like `Get-ChildItem`. For Rust development either works, but PowerShell is the modern default.

### 2. Why is the file named `main`?
The filename `main.rs` itself isn't magic — you can name a Rust source file anything. But the **function** named `main` inside it *is* special: it's the entry point Rust looks for when running an executable. By convention, the file containing `fn main()` is also called `main.rs`.

### 3. What is a Rust macro? Why does `println!` call one?
A **macro** is code that writes code. At compile time, the macro expands into actual Rust code before compilation continues. `println!` is a macro (not a function) because it accepts a variable number of arguments and does compile-time checks on the format string (e.g. `"{}"` placeholders) — regular Rust functions can't do that.

### 4. What's the difference between `println!` and `println`?
- `println!` (with `!`) is a macro — it's the real thing in Rust's standard library.
- `println` (without `!`) would be a regular function call. There is no `println` function in Rust; only the macro exists.
- The `!` is how Rust signals at the call site: "this is a macro, not a function."

### 5. What does "argument" / "passing to `println!`" mean?
An **argument** is a value you give to a function (or macro) when you call it. Inside `println!("Hello, world!")`, the string `"Hello, world!"` is the argument. "Passing it to `println!`" just means handing that value over so the macro can use it — in this case, print it.

### 6. What does the semicolon `;` mean — "this expression is over"?
Rust code is built from **expressions** (things that produce a value or perform an action). The `;` marks the end of one statement so Rust knows where it stops and the next one starts. Without it, Rust would think the next line is a continuation. Think of it like a period at the end of a sentence.

### 7. What does "compile" mean? What is the Rust compiler?
**Compiling** is translating human-readable source code (`.rs` files) into machine code that the computer's CPU can run directly. The **Rust compiler** is the program that does this translation — `rustc` is the command that invokes it.

### 8. What's the difference between `rustc` and `rustup`?
- `rustup` manages **Rust itself** — installing it, updating versions, switching toolchains. You use it rarely.
- `rustc` is the **Rust compiler** — it turns your `.rs` source files into executables. You use it (or Cargo, which uses it under the hood) every time you build code.
- So `rustc --version` asks the compiler what version it is; `rustup update` asks the version manager to fetch a newer compiler.

### 9. What is a "binary executable"? How is this different from Python? How is it similar to C/C++?
A **binary executable** is a file containing machine code the CPU can run directly — no translator needed at runtime.
- **Like C / C++**: you compile source code once into a standalone executable; it runs on its own.
- **Unlike Python**: Python doesn't produce a standalone binary by default. The Python interpreter reads your `.py` file and executes it line-by-line at runtime. So Python users need Python installed; Rust users just need the binary.

### 10. What are the `.exe` and `.pdb` files? Why are they produced?
- `main.exe` — the actual compiled executable (the binary you run).
- `main.pdb` — a "Program Database" file containing debug symbols (function names, line numbers, etc.). Useful for debuggers so they can show meaningful info if the program crashes. You can ship the `.exe` without the `.pdb`.
- They're produced because that's what compiling does on Windows: source `.rs` → executable `.exe` (+ debug info `.pdb`).

### 11. What is `.\main`? Why is there an extra step after `rustc`?
`.\main` tells PowerShell: "run the file named `main` in the current directory" (the `.\` prefix means "right here"). The two steps are separate because they do different things:
1. `rustc main.rs` — **compiles** the source into an executable. Done once.
2. `.\main` — **runs** the executable. Done as many times as you want.

This way you can recompile only when the source changes, and run the binary as often as you like without recompiling.

### 12. In Python you don't compile and run as separate steps — what does that mean?
When you run `python hello.py`, Python's interpreter reads the source file and executes it directly, in one step. There's no separate "compile" command and no executable file produced. With Rust you have two distinct phases: compile (`rustc`), then run (`.\main`).

### 13. What does "ahead-of-time compiled language" mean?
"Ahead-of-time" (AOT) means the code is compiled to machine code **before** you run it — not during. The compile step happens once, on your machine; afterwards anyone can run the resulting binary without recompiling.
- Contrast with **interpreted** languages (Python, Ruby), where source is translated as it runs.
- Contrast with **JIT-compiled** languages (Java, JavaScript with V8), where compilation happens *while* the program runs.

### 14. So `.py`, `.rb`, `.js` files have dependencies — what does that mean?
It means the recipient needs the corresponding runtime installed:
- a `.py` file needs Python on the recipient's machine,
- a `.rb` file needs Ruby,
- a `.js` file needs Node (or a browser).

With Rust, the compiled executable is self-contained — you ship the `.exe` and it just runs. That's the dependency tradeoff.

### 15. "Everything is a tradeoff in language design" — what does that mean?
Every choice a language makes gives up something to gain something else.
- **Rust** chooses AOT compilation → fast standalone binaries, but slower edit-run cycles and a more complex toolchain.
- **Python** chooses interpretation → instant feedback and easy scripting, but slower runtime and a runtime dependency.

There's no "best" language — only languages that prioritize different things (speed, safety, simplicity, portability, etc.). Picking a language means picking which tradeoffs you're okay with.
