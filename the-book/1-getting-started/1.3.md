# 1.3 — Hello, Cargo!

## What is Cargo?

**Cargo** is Rust's **build system** and **package manager**. It comes installed with Rust (you don't install it separately).

Most "Rustaceans" — developers who use, contribute to, or are passionate about Rust — use Cargo to manage their Rust projects, because Cargo handles a lot of tasks for you:

- **Building** your code (turning your `.rs` files into an executable).
- **Downloading the libraries** your code depends on.
- **Building those libraries** so your code can use them.

The libraries your code needs are called **dependencies**.

The simplest Rust programs (like the `hello_world` we wrote in 1.1) have no dependencies. If we had built `hello_world` with Cargo, Cargo would only have used the *building your code* part of itself — not the dependency-handling part. As you write more complex Rust programs you'll add dependencies, and starting your project with Cargo from the beginning makes adding them much easier later.

## Commands at a glance

```
cargo --version             # check that Cargo is installed
cargo new hello_cargo       # create a new Cargo project
cargo init                  # turn the current folder into a Cargo project
cargo build                 # compile the project (debug build)
cargo build --release       # compile with optimizations (for shipping)
cargo run                   # build + run in one step
cargo check                 # check it compiles, but don't produce an executable
```

## Creating a project with Cargo

```
cargo new hello_cargo
```

This creates a new directory and project called `hello_cargo`. Cargo creates the project's files inside a folder of the same name.

If you `ls` inside the new folder, you'll see:

- `Cargo.toml` — the project's configuration file.
- `src/` — a folder containing `main.rs` (your source code).

### Project layout — `src/`

Cargo expects your source files to live inside the `src/` directory. This is one of the ways using Cargo helps you organize your projects: source code in `src/`, config at the root, build output in `target/`.

### Converting an existing project to Cargo

If you started a project that doesn't use Cargo, you can convert it:

1. Move your project's source code into a `src/` directory.
2. Create a `Cargo.toml` file at the project root.

To create the `Cargo.toml` automatically, run:

```
cargo init
```

This turns the current folder into a Cargo project without creating a new subfolder.

### Git initialization

Cargo also initializes a new Git repository and creates a `.gitignore` file by default.

**Exception:** if you run `cargo new` *inside an existing Git repository*, Cargo will **not** create a new `.git` folder or `.gitignore` — because it assumes your project is already tracked by Git from the parent folder. (You can override this with `cargo new --vcs=git`.)

That's exactly why our `hello_cargo` didn't get its own `.git` folder — we ran the command inside the `D:\Rust` Git repo.

## The `Cargo.toml` file

Open `Cargo.toml` and you'll see something like:

```toml
[package]
name = "hello_cargo"
version = "0.1.0"
edition = "2024"

[dependencies]
```

Let's break it down.

### `[package]`

This is a **section header**. It says: "the lines below configure the package itself." As the project grows, you'll add other sections too.

Inside `[package]`:
- `name` — the name of your package.
- `version` — the current version of your package.
- `edition` — which edition of Rust to use (editions are big-batch language updates Rust ships every few years).

These three lines are the configuration information Cargo needs to compile your program.

### `[dependencies]`

Another section header. This is where you list the libraries your project depends on. In Rust, packages of code are referred to as **crates**.

Right now `[dependencies]` is empty, because `hello_cargo` doesn't depend on anything yet.

## Building and running with Cargo

### `cargo build`

```
cargo build
```

This compiles your project. It creates the executable at `target\debug\hello_cargo.exe` on Windows (not in your current directory).

Why `target\debug\`? Because the default build is a **debug build** — Cargo puts the binary inside a folder named `debug`. (See Q5 for what "debug build" means.)

You can run the executable manually with:

```
.\target\debug\hello_cargo.exe
```

### `Cargo.lock` (created on first build)

The first time you run `cargo build`, Cargo also creates a file at the top level called `Cargo.lock`. This file keeps track of the **exact versions** of every dependency in your project, so that future builds use the same versions.

You won't ever need to change this file manually — Cargo manages it for you.

### `cargo run`

```
cargo run
```

Compiles the code **and** runs the resulting executable, all in one command. This is more convenient than running `cargo build` and then typing the full path to the binary, so most developers use `cargo run`.

Example session:

```
PS D:\Rust\projects\hello_cargo> cargo build
   Compiling hello_cargo v0.1.0 (D:\Rust\projects\hello_cargo)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.88s
PS D:\Rust\projects\hello_cargo> .\target\debug\hello_cargo.exe
Hello, world!
PS D:\Rust\projects\hello_cargo> cargo run
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.01s
     Running `target\debug\hello_cargo.exe`
Hello, world!
```

Notice the second `cargo run` did **not** show "Compiling" — Cargo figured out on its own that the source hadn't changed, so it skipped rebuilding and just ran the binary.

If you *do* change the source code, Cargo will recompile:

```
PS D:\Rust\projects\hello_cargo> cargo run
   Compiling hello_cargo v0.1.0 (D:\Rust\projects\hello_cargo)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.41s
     Running `target\debug\hello_cargo.exe`
Hello, world!!
```

### `cargo check`

```
cargo check
```

This command quickly checks your code to make sure it compiles, but **does not produce an executable**.

```
PS D:\Rust\projects\hello_cargo> cargo check
    Checking hello_cargo v0.1.0 (D:\Rust\projects\hello_cargo)
    Finished `dev` profile [unoptimized + debuginfo] target(s) in 0.11s
```

`cargo check` is much faster than `cargo build` because it skips the step of producing an executable. You don't always need an executable — for example, while you're still writing the code and just want to confirm it compiles, `cargo check` is faster feedback.

**Habit to build:** run `cargo check` periodically as you write the program to make sure it still compiles. Run `cargo build` only when you're ready to use the executable.

### `cargo build --release` (for shipping)

When your project is finally ready for release, use:

```
cargo build --release
```

This compiles with **optimizations**, producing a faster executable in `target\release\` instead of `target\debug\`.

Tradeoff:
- Optimizations make your Rust code run faster.
- But turning them on **lengthens the time it takes to compile**.
- That's why you don't use `--release` while you're still developing — you want fast compile times during development, and only pay the optimization cost when shipping.

## Working with existing Cargo projects

Even though `hello_cargo` is simple, it already uses much of the real tooling you'll use in the rest of your Rust career. To work on any existing Rust project, you typically run:

```
git clone example.org/someproject
cd someproject
cargo build
```

That's it — Cargo reads the project's `Cargo.toml`, downloads any dependencies, and builds it.

## Questions & Doubts (cleared)

### 1. What does "build system" and "package manager" mean? Is this similar to Docker?

Two separate jobs, both bundled into Cargo:

- **Build system**: a tool that takes your source code and turns it into a runnable program. It runs the compiler (`rustc`) for you, in the right order, with the right flags. Without a build system, you'd have to type out every compile command by hand for every file.
- **Package manager**: a tool that downloads, installs, and keeps track of code written by other people that your project uses. Like `npm` for JavaScript, `pip` for Python, or `apt` on Linux.

**Is it like Docker?** No. Docker bundles your *whole running environment* (operating system, dependencies, your app) into a container so it runs the same anywhere. Cargo only deals with **building Rust code and managing Rust libraries**. Different layer of the stack.

### 2. What does "building your code" actually mean? Example?

"Building" = taking your source code and turning it into something the computer can run.

For Rust that means:
1. Reading your `.rs` files.
2. Running the compiler (`rustc`) on them.
3. Linking everything together.
4. Producing an executable file (e.g. `hello_cargo.exe`).

**Without Cargo** (what we did in 1.2):
```
rustc main.rs        # one file, one command, you do it manually
.\main
```

**With Cargo** (what we're doing now):
```
cargo build          # Cargo reads Cargo.toml, finds your code in src/,
                     # compiles it all, and puts the executable in target/
cargo run            # build + run in one step
```

Cargo handles the tedious parts (which files to compile, in what order, where to put the output) so you don't have to.

### 3. What is a library? How does it relate to this?

A **library** is reusable code someone else wrote, that you can plug into your project instead of writing everything from scratch.

For example, if you want to:
- generate random numbers → use the `rand` library
- make HTTP requests → use the `reqwest` library
- parse JSON → use the `serde_json` library

Without libraries, you'd have to write all of that yourself. With Cargo, you just add the library's name to `Cargo.toml` under `[dependencies]`, and Cargo downloads it, builds it, and lets you use it in your code.

In Rust, libraries are called **crates**.

### 4. "If we had built hello_world with Cargo, it would only use the part that handles building your code" — what does that mean?

Cargo has two main jobs:
1. Build your code.
2. Manage dependencies (download/build libraries).

`hello_world` from 1.1 had **no dependencies** — no external libraries. So if we had used Cargo for it, only job #1 (building) would have done any work. Job #2 (managing dependencies) would have had nothing to do because there were no dependencies to manage.

It's just saying: Cargo is still useful even on simple projects, but on simple projects you only see half of what it does.

### 5. What is a "debug build"? Why does Cargo put it in `target/debug/`?

When you run `cargo build`, Cargo produces a **debug build** by default. That means:

- It compiles **fast** (good while you're developing — quick feedback).
- It includes extra **debug information** so tools (and error messages) can show you exactly where things went wrong.
- It does **not optimize** the code, so the executable runs slower.

Cargo puts it in `target/debug/` so it's separate from the optimized **release** build (which goes in `target/release/` and is what you ship to users). Two folders, two purposes:

| Folder | When to use | Compile speed | Run speed |
|---|---|---|---|
| `target/debug/` | while developing (`cargo build`) | fast | slower |
| `target/release/` | when shipping (`cargo build --release`) | slow | fast |

### 6. What is the `.toml` extension?

`.toml` is just a file format — like `.json`, `.yaml`, or `.ini`. It stands for **Tom's Obvious, Minimal Language**. It's designed to be easy for humans to read and write.

A `.toml` file looks like this:
```toml
[section_name]
key = "value"
number = 42
```

Cargo uses `.toml` for `Cargo.toml` because it's clean and easy to edit by hand. You don't need to know anything more about TOML for now — just know it's a config file format.

### 7. What does "configuring a package" mean?

A **package** is a project that Cargo can build. "Configuring" the package means **telling Cargo the basic facts about it**, like:
- its name,
- its version,
- which Rust edition to use,
- what it depends on.

That's all `Cargo.toml` is — a settings file with answers to those questions. When you run `cargo build`, Cargo reads this file to know how to build your project.

### 8. What's the difference between a **package** and **dependencies**?

- **Package** = your project as a whole. The thing you're building. `hello_cargo` is a package. It has a name, a version, source code, and a `Cargo.toml`.
- **Dependencies** = other people's packages (libraries / crates) that *your* package needs in order to work.

Analogy: think of a recipe.
- Your recipe (the cake you're baking) = your **package**.
- The flour, sugar, and eggs you buy from the store = your **dependencies**.

You write your package. You *use* dependencies that other people wrote.
